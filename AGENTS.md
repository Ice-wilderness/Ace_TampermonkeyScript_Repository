# AGENTS.md

本仓库是自用油猴 / 用户脚本合集。每个 `.js` 脚本独立维护，README.md 和 CHANGELOG.md 均按脚本分区记录。

AI 在修改本仓库时，必须优先判断“当前改动属于哪个脚本”，再决定需要同步更新哪些文档和脚本头部元信息。

## 仓库结构约定

- 每个油猴脚本通常是一个独立的 `.js` 文件。
- `README.md` 用于说明脚本用途、主要功能、配置项、兼容页面、安装方式和维护说明。
- `CHANGELOG.md` 按脚本分区记录每个脚本的版本更新。
- 新增脚本时，必须同时更新 `README.md` 和 `CHANGELOG.md`。
- 修改已有脚本时，应只更新该脚本对应的 README 分区和 CHANGELOG 分区，不要无关改动其他脚本说明。

## 修改脚本时的文档规则

每次修改任意 `.js` 油猴脚本后，必须检查是否需要更新：

- `README.md`
- `CHANGELOG.md`
- 脚本文件头部的 userscript 元信息

尤其要检查以下 userscript 元信息是否仍然准确：

- `@name`
- `@description`
- `@version`
- `@match`
- `@include`
- `@grant`
- `@require`
- `@connect`
- `@license`

如果新增、删除或调整了脚本使用的 GM API，必须同步检查 `@grant`。

如果新增或改变了适配网站、页面路径、URL 规则，必须同步检查 `@match` / `@include`，并更新 `README.md` 的兼容页面说明。

## README.md 更新规则

当改动影响用户理解、安装或使用脚本时，必须更新 `README.md` 中对应脚本分区。

以下情况必须更新 `README.md`：

- 新增脚本。
- 删除脚本。
- 脚本改名或文件名变化。
- 新增、删除或明显改变主要功能。
- 新增、删除或改变配置项。
- 新增、删除或改变兼容页面。
- 改变数据存储结构、导入导出格式或迁移方式。
- 改变脚本使用方式、安装方式或注意事项。
- 修复的问题会影响 `README.md` 中现有描述的准确性。

以下情况通常不需要更新 `README.md`：

- 仅修复内部实现细节。
- 仅重构代码但不改变用户可见行为。
- 仅优化性能但不改变功能描述。
- 仅调整注释、日志或代码格式。
- 仅修复 `README.md` 中没有提到的边缘 bug。

如果 `README.md` 不需要更新，最终回复中应说明：

> README.md 未更新，因为本次改动不影响功能说明、配置项、兼容页面或使用方式。

## CHANGELOG.md 更新规则

每次修改任意 `.js` 脚本后，必须更新 `CHANGELOG.md` 中对应脚本分区。

本仓库的 `CHANGELOG.md` 按脚本分区维护，不使用统一的全局 `[Unreleased]` 分区。

更新规则：

1. 找到被修改脚本对应的分区。
2. 在该脚本分区顶部新增一个版本小节。
3. 版本号应与脚本文件头部 `@version` 保持一致。
4. 用简洁项目符号说明本次改动。
5. 不要把某个脚本的更新写到其他脚本分区。
6. 未修改的脚本分区不要新增“暂无本次更新”。

推荐格式：

```md
## 脚本名称

文件：[脚本文件名.js](./脚本文件名.js)

### x.y.z

- 修复……
- 新增……
- 优化……
````

如果当前改动涉及多个脚本，则每个被修改脚本都必须在 `CHANGELOG.md` 中新增对应版本记录。

## 版本号规则

修改油猴脚本时，必须检查并更新脚本头部的 `@version`。

建议采用语义化版本：

* 修复 bug：递增补丁版本，例如 `3.1.14` -> `3.1.15`
* 新增功能或兼容新页面：递增次版本，例如 `3.1.14` -> `3.2.0`
* 大幅重构、删除功能、数据结构不兼容：递增主版本，例如 `3.1.14` -> `4.0.0`

如果只是修改 `README.md`、`CHANGELOG.md`、`AGENTS.md` 或注释，不需要更新脚本版本号。

## 新增脚本规则

新增 `.js` 油猴脚本时，必须同时完成：

* 在 `README.md` 的“脚本列表”中加入链接。
* 在 `README.md` 中新增该脚本分区。
* 说明脚本文件名、用途、主要功能、配置项和适配页面。
* 在 `CHANGELOG.md` 中新增该脚本分区。
* 在 `CHANGELOG.md` 中记录初始版本。
* 检查脚本头部 userscript 元信息是否完整。

## 删除或重命名脚本规则

删除脚本时：

* 从 `README.md` 的脚本列表中移除。
* 删除或调整 `README.md` 中对应脚本分区。
* 在 `CHANGELOG.md` 对应分区记录删除原因，或在仓库文档中说明该脚本已移除。
* 检查是否有其他文档仍链接到旧文件名。

重命名脚本时：

* 更新 `README.md` 中的文件链接。
* 更新 `CHANGELOG.md` 中的文件链接。
* 检查脚本头部 `@name` 是否需要同步调整。
* 确保旧文件名不会继续出现在文档链接中。

## 代码修改注意事项

修改油猴脚本时，应尽量保持：

* 不破坏已有用户数据。
* 不随意更改 GM 存储 key。
* 不随意删除导入、导出、备份或恢复逻辑。
* 不引入不必要的外部依赖。
* 不扩大 `@match` 范围，除非本次任务明确需要。
* 不增加多余权限，尤其是 `@grant`、`@connect`。
* 对 SPA 页面、懒加载内容、滚动加载内容的处理要谨慎，避免重复绑定事件或重复插入元素。
* 对已有选择器、MutationObserver、定时器和缓存逻辑的修改要注意性能影响。

## 提交前检查

在完成任务前，必须检查：

```bash
git diff -- README.md CHANGELOG.md
git diff -- '*.js'
```

如果修改了脚本，还必须确认：

* 对应脚本的 `@version` 已更新，或明确说明为什么不需要更新。
* `CHANGELOG.md` 中对应脚本分区已记录本次改动。
* `README.md` 是否需要更新已经判断过。
* userscript 头部元信息仍然准确。
* 没有无关格式化或无关脚本改动。

## 最终回复要求

完成任务后，最终回复应包含：

* 修改了哪些脚本。
* 是否更新了 `README.md`。
* 是否更新了 `CHANGELOG.md`。
* 是否更新了脚本 `@version`。
* 如果 `README.md` 未更新，说明原因。
* 如果未运行测试或无法验证，应明确说明。
